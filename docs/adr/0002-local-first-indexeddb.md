# ADR 0002: ローカルファースト (IndexedDB) — サーバDB・認証を持たない

- ステータス: Accepted
- 日付: 2026-07-10

## コンテキスト

スナップ画像・手持ち服・好みデータをどこに保存するか。スターター既定は Supabase (Postgres + Auth + Storage)。
一方このアプリの価値は「撮った瞬間の体感速度」と「登録の摩擦の低さ」で決まり、
MVP 時点で複数端末同期の要求はない。服・体型に関わるデータはプライバシー感度も高い。

## 検討した選択肢

1. **Supabase (クラウド)**: 同期・アカウントが得られるが、認証画面/RLS設計/Storage設定でMVPが重くなり、初回利用に登録の壁ができる
2. **IndexedDB ローカルファースト (採用)**: 認証ゼロ・即利用開始・オフライン閲覧可・プライバシー露出最小。同期は将来の追加レイヤーにする

## 決定

- Dexie (IndexedDB) に全データを保存する。サーバDB・認証・サーバ状態を持たない
- 画像は **Blob のまま**保存する (base64 化はサイズ1.33倍と GC 負荷のため禁止)
- UI は `useLiveQuery` で DB に追従し、書き込みは楽観的 (シャッター直後に `status: analyzing` で保存)
- サーバは AI 中継のみ (`/api/*` はステートレス)

## 帰結

- (+) 初回利用の摩擦ゼロ、オフライン閲覧、AI リクエスト以外の外部送信なし
- (+) サーバ運用コスト・障害点がほぼゼロ
- (−) 端末を変えるとデータが消える。ブラウザのストレージ削除で消える
- (−) likedPoints 等の集計は毎回クライアント導出 (件数が増えたら要見直し)

## 再評価する条件

- 複数端末同期・バックアップの要望が出た場合 → Supabase 等を「同期先」として追加 (置換ではなく)
- クローゼットが数百件規模になり集計が遅くなった場合
