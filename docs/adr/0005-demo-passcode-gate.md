# ADR 0005: デモゲート — 共有パスコード + httpOnly Cookie (ユーザー認証は導入しない)

- ステータス: Accepted
- 日付: 2026-07-10

## コンテキスト

本番 URL (https://speadwearneo.vercel.app) を公開デモとして配り始めたが、
URL を知っていれば誰でも AI API (`/api/analyze` 等) を叩ける状態は、実キー設定後に
無制限課金の入口になる。一方、本プロジェクトは「サーバ側 DB・認証を導入しない」を
禁止事項にしており (CLAUDE.md / ADR-0002)、ユーザーアカウントは作りたくない。
必要なのは「見せたい相手だけに絞る」入場ゲートであって、認証・認可の体系ではない。

## 検討した選択肢

1. **Vercel Deployment Protection**: Password Protection は有料アドオン。無料の Standard
   Protection は preview のみで本番エイリアスを守れない
2. **クライアント側ゲート (localStorage)**: 実装最小だが JS を読めば迂回でき、API は無防備のまま
3. **共有パスコード + httpOnly Cookie (採用)**: `proxy.ts` で全ページ・全 API をサーバ側判定。
   DB 不要・アカウント不要で、環境変数 1 つで有効/無効を切り替えられる

## 決定

- `DEMO_PASSCODE` (環境変数) が**設定されている環境でのみ**ゲートが有効になる。
  未設定 (ローカル開発・CI・モック E2E の既定) では従来どおり全公開で動き、テストに影響しない
- `src/proxy.ts` (Next.js 16 の proxy 規約 — 旧 middleware) が全ルートを判定:
  Cookie 不一致のページは `/login` へリダイレクト、API は 401 JSON
- Cookie 値は生パスコードではなく `SHA-256(salt + passcode)` の hex (`src/lib/demo-gate.ts`)。
  検証 API (`POST /api/login`) は導出ハッシュ同士を `timingSafeEqual` で比較する
- 素通しはログイン導線 (`/login`, `/api/login`) と PWA アセット (manifest / アイコン / favicon) のみ
- これは**ユーザー認証ではない**: アカウント・セッション・ユーザーIDは存在せず、
  データは引き続き各端末の IndexedDB のみ (ADR-0002 のローカルファーストは不変)

## 帰結

- (+) デモ URL の配布範囲を制御でき、実キー設定後の野良課金リスクを塞げる
- (+) DB・認証基盤なし。パスコード変更 = 環境変数変更 + 再デプロイのみ (ソルトのバージョンで旧 Cookie 一括失効も可能)
- (−) パスコードは共有秘密であり、漏れたら配り直し。個人単位の失効はできない
- (−) 総当たり対策 (レートリミット) は未実装。桁数の長いパスコード運用で緩和し、
  サービス公開時はレートリミット導入と本 ADR の置き換えを必須とする

## 再評価する条件

- 一般公開 (不特定多数) に切り替える場合 → ゲート撤去 + API レートリミット導入
- ユーザーごとのデータ同期・アカウントが要件になった場合 → 本 ADR を破棄し認証 ADR を新規作成
